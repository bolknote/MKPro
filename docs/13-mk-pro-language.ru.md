# Язык MK-Pro

MK-Pro описывает компактную программу для калькулятора человеческими терминами:
состояние, ввод, вывод на дисплей, циклы и функции. Автору не нужно вручную
включать тёмные входы, трюки с X2, наложения или недокументированные опкоды.
Это возможности машины MK-61, и оптимизатор может применять их всякий раз,
когда проверки эффектов и локальные статические доказательства разрешают
переписывание.

Исполняемые примеры в верхнеуровневом каталоге `examples/*.mkpro` намеренно
написаны на этом человекоориентированном DSL V2. Исходник MK-Pro состоит из
одного блока `program` и необязательных метаданных отчёта; отдельный синтаксис
сырых калькуляторных листингов не входит в язык.

## Форма

```mkpro
program CounterGame {
  state {
    score: counter 0..9 = 0
    food: counter 0..9 = 5
  }

  fn main_screen() {
    show(score, food)
  }

  fn game_over() {
    halt(0)
  }

  loop {
    main_screen()
    key = read()

    match key {
      2 => gain()
      8 => spend()
      otherwise => game_over()
    }
  }

  fn gain() {
    score++
    main_screen()
  }

  fn spend() {
    food--
    main_screen()
  }
}
```

## Без подсказок

В MK-Pro нет синтаксиса подсказок в скобках. Если слово не меняет сгенерированную
программу, валидацию или решения оптимизатора, ему место в комментарии, а не в
языке. Компилятор сам владеет такими реализационными решениями, как X2, тёмные
входы, наложения, размещение регистров, байты дисплея и недокументированные
опкоды.

Современная поверхность парсера использует только актуальные формы команд:
`show(...)`, `preview(...)`, `halt(...)`, `name = read()`,
`if/unless/while/match`, `loop`, `raw` и блоки `fn`. Старые командообразные
исходные формы вроде `read X`, `show X`, `stop X`, `move X <dir>`,
`screen ...`, `turn ...`, `rule ...`, `challenge ...`, `fleet ...`,
`encounters ...` или `world ...` не являются отдельной поверхностью парсера:
они падают как обычный неверный синтаксис, а не сохраняются как legacy-ключевые
слова.

## Комментарии

MK-Pro принимает построчные и хвостовые комментарии как через `//`, так и через
`#`:

```mkpro
// A whole-line comment.
key = read()  # A trailing comment.
```

## Верхний уровень

Вне `program` допустимы только необязательные метаданные отчёта:

- `reference name`

Смысл игры должен находиться внутри `program`. Счётчики, наборы клеток, карты,
события, случайность и факты о packed-таблицах должны выражаться обычными
декларациями, `state`, циклами, `match` и функциями. Отчёт компилятора показывает,
какие регистры, наложения, setup-константы, hex-мантиссы, инициализация
случайности или другие реализационные тактики были выбраны.

## Блоки программы

Текущая человекоориентированная DSL-поверхность внутри `program` такова:

- `state { ... }` для игрового состояния, счётчиков, масок, координат и
  scratch-полей.
- `name: board(0..9, 0..9)` для фиксированных координатных доменов доски.
- `name: board(encoding)` для компактных генерируемых координатных доменов.
- `loop { ... }` для основного игрового цикла и `fn name(...) { ... }` для
  именованных действий.
- `name = read()` внутри цикла или функции для значения, которое вводит игрок.

Функции должны оставаться на уровне игровых действий: `player = player + 10`,
`safe_landing()`, обычные присваивания, сравнения, диспетчеризация и остановы.
Понижатель превращает их в присваивания, команды дисплея, диспетчеризацию и
остановы.

Переменные берутся из полей `state { ... }`, привязок досок/доменов и параметров
функций. Присваивание необъявленному имени всё ещё работает — компилятор
неявно выделяет регистр, — но выдаёт предупреждение, потому что на машине с 15
регистрами опечатка незаметно съедает один из них. Цели `name = read()`
повышаются до scratch-полей состояния без предупреждения. Флаг CLI `--strict`
(`strictAllocation` в API) превращает любое неявное выделение в ошибку, включая
молчаливое повышение `read()`: строгие программы должны объявлять каждую
переменную явно.

В состоянии можно объявить `expected_mode("rad"|"deg"|"grd")`, чтобы на этапе
setup сгенерировать проверку режима углов. Используйте `expected_mode_only(...)`,
когда исходник также обещает, что переключатель Р-ГРД-Г не будет меняться во
время выполнения программы. Эта более сильная форма включает runtime-переписывания
тригонометрии, результат которых зависит от выбранных единиц угла.

Вывод на дисплей — это список видимых фрагментов. Фрагментом может быть состояние
или текст:

```mkpro
state {
  bottles: counter 0..99 = stack.X
}

fn beer() {
  show("BEEr ", bottles:02)
}
```

Запятые разделяют фрагменты; они не добавляют видимых символов. `show(a, b)`
означает `a` сразу за которым идёт `b`; для явного пробела пишите
`show(a, " ", b)`, а для `a-b` — `show(a, "-", b)`. Соседние фрагменты без
запятых являются синтаксической ошибкой. Исходник может запросить фиксированную
ширину с заполнением нулями, например `bottles:02` или `score:03`; без явной
ширины естественную видимую ширину задаёт диапазон счётчика (`0..99` — две
цифры, `0..9999` — четыре). Компилятор сам решает, станет ли такой вывод
обычным числовым выводом, packed-байтами дисплея, знаково-цифровыми формами,
`К ИНВ` или другим доказанным понижением MK-61.

Текстовые фрагменты в `show(...)` используют алфавит дисплея MK-61: цифры,
пробелы, `-`, `L`, `С`, `Г` и `Е` (латинские `C`, `D`/`G`, `E`, `O` и
кириллические похожие символы нормализуются там, где они соответствуют ячейкам
калькулятора). Строки предназначены только для дисплея; они не являются
runtime-строковыми значениями. Временное присваивание вроде `clue = "-"`
протягивается в следующий `show(...)` только когда компилятор может доказать,
что значение используется как фрагмент дисплея. Это позволяет писать такие
шаблоны:

```mkpro
clue = "-"
show(room, " ", clue)
```

и всё равно понижать их как display-шаблон `show(room, " -")`, не сохраняя
`"-"` в числовом регистре.

Понижение сначала проверяет короткие специальные случаи (`ЕГГОГ`, готовые
литеральные видео-строки, трюки со знаком/экспонентой и другие известные
компактные формы). Если специальный случай не подходит, общий строитель
display-ячеек обрабатывает смешанные экраны фиксированной ширины до восьми
display-ячеек, включая комбинации числовых полей и литеральных ячеек, такие как
`show(room, " ", arrows, " ", clue)`, `show("L-L ", clue)` и `show("0 L", clue)`.
Начальные и конечные литеральные пробелы обрезаются, потому что у дисплея
калькулятора нет стабильного видимого краевого пробела.

Packed-видеостроку также можно встроить под однозначное ведущее поле через
`show(floor, ".", row)`, когда `row` — значение `packed`, уже содержащее семь
дробных display-ячеек. Это соответствует исходным листингам, которые сначала
собирают картинку комнаты, а затем вставляют текущий этаж через `↔; F↻; ВП`.

`show(...)` — возобновляемый видимый останов. Используйте его для экранов, после
которых игрок продолжает программу, часто с последующим `name = read()`. Для
терминальных экранов помещайте видимое значение прямо в `halt(...)`.

`show("ЕГГОГ")` понижается как возобновляемая пауза ошибки: компилятор эмитит
одноклеточный opcode ошибки плюс пропущенную padding-ячейку, потому что
калькулятор продолжает выполнение с ячейки после этого padding. `halt("ЕГГОГ")`
может использовать только одноклеточный opcode ошибки, потому что терминальный
путь исходного уровня не имеет возобновляемого продолжения.

`preview(expr)` вычисляет числовое выражение и оставляет его видимым без эмита
`С/П`. Используйте его для исходных листингов, которые намеренно кратко
показывают или подготавливают текущее display-значение перед следующим реальным
остановом.

Не записывайте setup- или storage-тактики как верхнеуровневые реализационные
блоки:

```mkpro
preload R9 = 0.5
table packed code_overlay giant_country_tables { floor_plans may_overlay address_cells }
```

MK-Pro хранит реальные игровые факты в игровом языке:

```mkpro
strength: counter 0..99 = 40

plans: cells(giant_country) = random()
```

## Константы времени компиляции

Фиксированные числовые значения, которые никогда не меняются во время выполнения,
объявляйте через `const` на уровне программы (рядом с `state`, досками и
функциями):

```mkpro
program RamboIII {
  const REINFORCEMENT_CAP = 10000
  const VICTORY = 8.1020088E14

  state {
    scratch: packed
  }

  fn won() {
    halt(VICTORY)
  }

  fn reinforce() {
    scratch = int(random_state * REINFORCEMENT_CAP)
  }
}
```

Правила:

- Правая часть должна быть **числовым выражением времени компиляции**:
  литералы, ссылки на более ранние имена `const` в той же программе и `+ - * /`
  (с унарным `-`). Вызовы вроде `random()`, чтения и имена состояния
  запрещены.
- Именам `const` нельзя присваивать значения; они раскрываются inline во всех
  местах использования.
- `const` не может повторять имя поля `state`.

Компилятор подставляет каждый `const` в места использования до распределения
регистров и свёртки констант, так что имя никогда не занимает регистр состояния.
Существующий pass `preloaded-constant` всё ещё решает, дешевле ли повторяющийся
литерал как inline-последовательность цифр или как общий предзагруженный
регистр. Используйте `const` для именованных скаляров в выражениях
(`halt(...)`, `int(x * CAP)`, условия). Используйте indexed initializer list
(см. ниже), когда значение является **начальным содержимым ячейки банка
состояния** на этапе setup.

## Конфигурация состояния

Поля состояния несут только данные, влияющие на программу:

```mkpro
state {
  strength: counter 0..99 = 40
  score: counter 0..99 = 0

  plans: cells(giant_country) = random()
}
```

Используйте `counter` для ограниченных числовых значений, включая расходники,
очки, топливо и счётчики оставшихся фигур. Используйте `coord(domain)` для
координаты клетки, `cells(domain)` для сгенерированной карты, мин, стен или
масок столкновений, а `coord_list(domain, count)` — для списка координат
фиксированного размера на доске. Полный канонический набор типов состояния:
`flag`, `counter`, `coord(domain)`, `cells(domain)`,
`coord_list(domain, count)` и `packed`. Размещение регистров, хранение адресов
и компактная упаковка — решения компилятора.

Используйте `packed` для обычного числового хранения, которое не является
ограниченным счётчиком: резервы, случайные seed-значения, scratch-поля,
терминальные видео-значения и другие калькуляторные числа, которые могут
содержать десятичные части или большие порядки. `counter` несёт видимый
диапазон цифр; `packed` — нет.

```mkpro
state {
  reserve: packed = 25000
  random_state: packed = 0.5
  scratch: packed = 0
}
```

Некоторые оригинальные игры MK-61 хранят несколько связанных фактов в одном
регистре как одно десятичное значение, например `5000.999` для силы и
дистанции вместе. Выражайте это как одно поле `packed` с тем же литералом:
компилятор хранит значение, а не отдельную игровую модель.

### Индексированные банки состояния

Когда несколько параллельных массивов используют один и тот же диапазон
индексов, объявляйте их как группу состояния:

```mkpro
state {
  front_line: group(1..3) {
    front: packed = 5000.999
    robots: packed = 0
  }
}
```

Читайте и записывайте поля группы через индексированный доступ:

```mkpro
front = front_line[slot].front
robots = front_line[slot].robots
front_line[slot].front += scratch
front_line[2].robots = 0
```

Диапазон группы включительный. Все поля одной группы должны использовать один и
тот же диапазон. Вложенные группы не поддерживаются, и поле группы не может
одновременно объявлять собственный диапазон индексов `[min..max]`.

Для одного индексированного массива `packed` скобочная форма эквивалентна:

```mkpro
state {
  slots: packed[1..3] = 0
}

loop {
  slots[i] = x
  halt(slots[2])
}
```

Индексированные массивы могут использовать список инициализаторов по элементам,
когда опубликованный setup загружает разные константы в смежный банк регистров:

```mkpro
state {
  walls: packed[1..3] = [bit_or(1.0080808, 1.7062264), bit_or(1.808, 1.6715401), bit_or(1.8088088, 1.2260335)]
}
```

Длина списка должна совпадать с включительным диапазоном банка. Каждый элемент
понижается как начальное значение соответствующей ячейки банка, поэтому
`walls[1]`, `walls[2]` и `walls[3]` остаются обычным индексированным состоянием
во время выполнения. Элементы списка также могут использовать `stack.X`,
`stack.Y` или `stack.X2`, когда один элемент банка вручную вводится до запуска
setup-программы.

Обе формы понижаются в смежные регистры калькулятора. Константный индекс вроде
`front_line[2].front` или `slots[2]` разрешается во время компиляции.
Динамический индекс вроде `front_line[slot].front` понижается через команды
косвенной памяти MK-61 (`К X->П` / `К П->X`) с compiler-owned регистром
селектора. Цели индексированного присваивания принимают те же индексные
выражения, что и индексированные чтения, включая контекстные координатные
helpers вроде `resources[pos.floor]`.

Когда индексное выражение является аффинной формой вроде `physical - 3` и это
значение уже называет физический регистр калькулятора для выбранного элемента
смежного банка, компилятор может использовать саму индексную переменную как
косвенный селектор вместо материализации отдельного селектора банка.

То же прямое переиспользование селектора применяется к `bank[int(selector)]`,
когда банк физически выровнен со своими логическими индексами, а `selector`
удерживается в `R7..Re`: косвенная адресация памяти MK-61 игнорирует дробный
хвост, поэтому packed-координата вроде `3.0000008` может напрямую выбрать
регистр `R3`.

Когда два динамических доступа используют один и тот же банк и одно и то же
индексное выражение, понижатель сохраняет первый селектор живым. Если у
соседнего поля другое смежное смещение и вывести его из закешированного селектора
дешевле, чем заново вычислять индекс, компилятор вспоминает закешированный
селектор, применяет разницу смещений и сохраняет селектор соседнего поля.

Когда три или более смежных индексированных поля имеют один и тот же
нелитеральный инициализатор, генерация setup может эмитить один косвенный цикл
заполнения через `R0` вместо копирования инициализатора для каждого элемента.
Это компактная setup-форма для сгенерированных таблиц строк, например девяти
packed-строк этажей в Treasure Hunter 2.

Когда цикл записывает через бегущий указатель и затем увеличивает его, например
`slots[pointer + 1] = value` с последующим `pointer++`, компилятор может слить
эту пару в одну косвенную запись с preincrement, если раскладка банка это
позволяет.

Начальные значения могут приходить из стартовых стековых/экранных регистров
`stack.X`, `stack.Y` и `stack.X2` — для игр, где игрок вводит setup-значения
перед запуском программы. `stack.X2` восстанавливает скрытое экранное значение
X2 через `.` во время setup, поэтому оно считывается до числовых setup-литералов,
которые сбили бы X2-контекст:

```mkpro
  state {
    pos: coord(cave) = stack.X
    food: counter 0..99 = stack.Y
    seed: counter 0..99 = stack.X2
  }
```

Это стартовая договорённость, а не общее связывание с регистрами. Регистры
памяти `R0`..`Re` распределяются компилятором; исходник не должен называть их
для размещения состояния.

Для запусков в эмуляторе и регрессионных тестов используйте setup-блок
компилятора вместо ручной записи присваиваний регистрам в исходнике. Исходник
сохраняет естественные факты состояния выше; `mk-pro explain` сообщает
выделенные регистры и готовый для эмулятора блок, например:

```text
`R3=0; R0=0; Rc=20`
```

Этот блок — сгенерированный вывод. Он полезен для загрузки или сравнения
скомпилированных программ, но не должен заменять высокоуровневые объявления
состояния.

Имена, начинающиеся с `__mkpro_`, зарезервированы для внутренних helper-выражений
компилятора. Пользовательские программы должны выражать игровые правила
напрямую; сгенерированные helper-имена могут появляться в отчётах, но они не
являются частью исходного языка.

## Терминальные функции

Терминальные исходы — обычные функции. Поместите финальную последовательность
display/halt в именованную функцию и вызывайте её как любое другое действие:

```mkpro
fn safe_landing() {
  halt(777)
}

fn crash() {
  halt(666)
}

fn touchdown() {
  if abs(speed) <= 5 {
    safe_landing()
  } else {
    crash()
  }
}
```

Многоходовые условия используют `else if`; это эквивалентно ветке `else`,
содержащей ещё один `if`, но делает исходник более плоским:

```mkpro
fn teleport_to_vault() {
  if charges >= 10 {
    charges -= 10
    pos = random(station)
    resolve_room()
  } else if charges >= 5 {
    charges -= 5
    pos = random(station)
    resolve_room()
  } else {
    caught()
  }
}
```

Используйте `halt(value)`, когда терминальный дисплей — значение, и
`halt("text")` для терминального калькуляторного видеолитерала вроде
`halt("ЕГГОГ")` или `halt("8СГ-Е-78")`. Не пишите `show(...); halt()` ради
одного финального экрана: это описывает два эффекта исходного уровня, даже если
конкретное понижение умеет сливать некоторые случаи. Компилятор проверяет это
как style lint: `show(...)` сразу перед голым `halt()` добавляет заметку в
`report.warnings`. Одноразовые терминальные функции inline-ятся, а общие
терминальные функции оптимизатор может понижать как прямые переходы, поэтому
авторам не нужна отдельная терминальная форма ради раскладки. Имена функций
должны быть уникальны, и вызов должен ссылаться на объявленную функцию.

## Функции, возвращающие значение (`return`)

`fn`, тело которой содержит `return expr`, — это **функция, возвращающая
значение**: она вычисляет значение и отдаёт его вызывающему коду. Такие функции
можно использовать в позиции выражения, в том числе вложенными в более крупные
выражения и внутри других функций.

```mkpro
fn square(n) {
  return n * n
}

fn sum_of_squares(a, b) {
  return square(a) + square(b)
}

loop {
  x = read()
  y = read()
  result = sum_of_squares(x, y)
  halt(result)
}
```

Семантика и правила:

- Возвращаемое значение передаётся в регистре X калькулятора. Вызов понижается
  в `ПП` (вызов подпрограммы); тело функции завершается `В/О`, который
  возвращает управление вызывающему коду с результатом в X. Затем вызывающий код
  потребляет X напрямую (например, `result = f(x)` сохраняет X в `result`, а
  `a + f(x)` прибавляет X).
- Функция, возвращающая значение, должна выполнять `return` на **каждом** пути
  управления. Завершайте каждую ветку `if`/`match` через `return` (или `halt`);
  функция, которая может дойти до конца тела, — ошибка компиляции.
- `return` допустим только внутри `fn`. Использование его в блоке `loop` или в
  основном теле — ошибка компиляции.
- Хвостовая рекурсия поддерживается. Рекурсивный цикл принимается только когда
  каждый вызов в цикле является всем выражением `return other_fn(...)`;
  компилятор понижает такой хвостовой вызов в `БП`, поэтому он не потребляет ещё
  один frame стека возврата. Нехвостовая рекурсия, например `return f(n) + 1`,
  запрещена, потому что MK-61 хранит только пять уровней возвратов подпрограмм.
- Вызов функции, возвращающей значение, как обычного statement разрешён; её
  результат просто отбрасывается.

Вложенные вызовы обрабатываются автоматически. Поскольку `ПП` затирает весь
рабочий стек X/Y/Z/T, вызов, вложенный в более крупное выражение (например
операнды `square(a)` выше), поднимается во временное значение с коротким временем
жизни до построения окружающего выражения, так что частично вычисленное значение
никогда не уничтожается. Оптимизатор освобождает эти временные значения, когда
может.

## Доски и наборы клеток

Используйте `board`, когда игра строится вокруг проб, выстрелов, фигур или
сгенерированных доменов движения. Прямоугольные координаты доски задаются
диапазонами, а не отдельным мини-синтаксисом `10x10`:

```mkpro
ocean: board(0..9, 0..9)

state {
  player_shot: coord(ocean)
  enemy_fleet: cells(ocean) = random()
  enemy_ships: counter 0..99 = stack.X
}
```

`board` описывает координатную систему. `cells(domain)` описывает сгенерированный
набор занятых клеток. Счётчики, считающие оставшиеся фигуры, — обычные поля
`counter`, а не скрытые сущности внутри специального блока. Это не заставляет
игры вроде Морского боя, Сапёра, охоты на лис или настольных головоломок
притворяться играми с коридорным движением, но позволяет им использовать ту же
форму деклараций, что и другое состояние.

Используйте `coord_list(domain, count)`, когда игре нужно точное число различных
координат доски, а не bitmask. Fox Hunt на MK-61 — эталонная форма:

```mkpro
field: board(0..9, 0..9)

state {
  cell: coord(field)
  foxes: coord_list(field, 9) = random_unique()
  bearing: counter 0..9 = 0
}

loop {
  cell = read()

  if cell in foxes {
    found_fox()
  }

  bearing = line_count(foxes, cell)
  show("--", cell:02, "--", bearing)
}
```

Имя списка — игровое состояние, видимое на уровне исходника. Компилятор понижает
его в смежные координатные регистры и внутренне использует
`coord_list_has(...)` и `coord_list_line_count(...)` для membership-проверок и
сканов строк/столбцов/диагоналей в стиле охоты на лис. Начальные значения:
`random_unique()` или `0`. `random_unique()` — startup-инициализатор, а не вызов
`random()`: компилятор заполняет список `count` уникальными координатами домена
доски, повторяя draws до отсутствия коллизий. Предпочитайте его
`cells(domain) = random()`, когда игре нужно известное число фигур без
пересечений. См. `examples/fox-hunt-mk61.mkpro` и сравните с
`examples/fox-hunt-100.mkpro`, где используется случайная cell-set mask.

Запросы к доске должны прямо называть геометрическую операцию:

```mkpro
fn scan_foxes() {
  bearing = line_count(foxes, cell)
}

fn reveal_safe_cell() {
  clue = neighbor_count(mines, probe)
}
```

`line_count(cells, cell)` — счётчик строки/столбца/диагонали в стиле охоты на
лис. `neighbor_count(cells, cell)` — счётчик восьми соседей в стиле Сапёра. Они
понижаются в обычные выражения с масками/цифрами, после чего оптимизатор может
выбрать общую арифметику, маски, десятичные цифры или query-tail. Для одномерных
досок, помещающихся в мантиссу, проверки принадлежности и обновления
`cells(domain)` понижаются
в десятичные позиционные маски вроде `pow10(cell)` вместо общей формулы
четырёхбитной маски.

## Домены досок

`board` объявляет координатный домен. Прямоугольные доски используют явные
диапазоны:

```mkpro
field: board(0..9, 0..9)
```

Компактные генерируемые доски используют имя кодирования:

```mkpro
giant_country: board(decimal_player)
```

Это семантическое обязательство, а не подсказка хранения: если у компилятора нет
понижателя для операции над такой доской, он отклоняет программу вместо выбора
готового игрового шаблона. Текущие компактные board-encoding'и:
`corridor_plan`, `decimal_player`, `floor_plan`, `packed_decimal_zero_run`,
`pier_to_ship` и `row_scan`.

Движение — обычная координатная арифметика над packed-позицией. Выберите delta,
соответствующую кодированию доски, и прибавьте её к координате:

```mkpro
fn move_south() {
  player = player + 1
  cave_screen()
}

fn move_dir(dir) {
  next = player + dir
  if next in walls {
    blocked = next
    show(0)
  }
  else {
    player = next
  }
}
```

Обычные grid-доски (`row_scan`) используют целочисленные delta: `+1`/`-1`
сдвигают разряд единиц, `+10`/`-10` — разряд десятков, `+100`/`-100` — разряд
сотен. Кодирование `packed_decimal_zero_run` упаковывает координату как
десятичное число с zero-run spacing, поэтому его ходы — маленькие дробные delta
(например `+0.0000002`). Используйте обычную форму update `pos += expr`, когда
величина движения вычисляется. Если функции нужно запомнить заблокированное
место назначения, запишите это как обычное состояние: `next = player + dir`, а
затем присвойте `blocked = next` только в ветке, где действительно встретилась
стена.

Запросы к доске используют ту же позицию выражения, что и обычные формулы:

```mkpro
fn inspect_cell() {
  tile = cell_at(cave, pos)
}

fn move_monster() {
  threat = random(harbor)
}
```

`cell_at(board, pos)` читает сгенерированный tile/event code для позиции на
доске. `random(board)` выбирает случайную координату из домена доски.

## Формулы

Формулы V2 используют тот же парсер выражений, что и конвейер компилятора.
Десятичные константы допустимы внутри более крупных формул, поэтому числовые
игры могут писать:

```mkpro
accel = burn * 10 / fuel - 9.8
```

Компилятор проверяет, что выражение можно понизить, до входа в генерацию кода.
Он больше не отклоняет формулы только из-за того, что десятичный литерал стоит
рядом с переменными или операторами.

Вспомогательные функции формул дают доступ к полезным калькуляторным операциям
без raw-опкодов:

```mkpro
height = pow(base, exponent)
probe = bit_has(walls, 5)
walls = bit_set(walls, 5)
walls = bit_clear(walls, 5)
walls = bit_toggle(walls, 5)

probe = cell_has(walls, x, y)
walls = cell_set(walls, x, y)
walls = cell_clear(walls, x, y)
walls = cell_toggle(walls, x, y)

code = digit_at(display_value, 2)
display_value = digit_add(display_value, 1, 7)
display_value = digit_set(display_value, 2, 9)

near = near_any(room, 1, pit1, pit2)
hit = eq_any(room, bat1, bat2)

roll = int(random(0, 10))
shot = int(random(0, 100))
```

`random()` понижается в псевдослучайную команду MK-61 (`К СЧ`) и возвращает
единичную дробь в `[0, 1)`. Команда никогда не выдаёт `1`. `0` может появиться
только в тех же редких случаях hexadecimal-`Y` stack, что и у базовой команды
калькулятора. Используйте её в инициализаторах состояния вроде
`cells(domain) = random()` и в формулах, которые масштабируют draw, например
`int(random() * 9) + 1`.

`random(max)` — синтаксический сахар для `random() * max`.
`random(min, max)` — синтаксический сахар для
`min + random() * (max - min)`. `random(domain)` выбирает координату из
board-домена. Числовая верхняя граница исключительная; оборачивайте вызов в
`int(...)`, когда числовому диапазону нужно целое число. `int(random(10))`
даёт `0..9`, `int(random(0, 10))` тоже даёт `0..9`, а
`int(random(0, 100))` даёт `0..99`. Дробные границы разрешены, когда нужен
непрерывный draw, например `int(random(0.5, 2.5))`. Компилятор берёт floor для
integer random draws через `x - frac(x)`, поэтому эти формы избегают opcode
MK-61 `К [x]` сразу после `К СЧ`.

`entered()` потребляет текущее введённое с клавиатуры значение X без эмита нового
`С/П`. Используйте его только сразу после существующего останова, когда
опубликованный UI MK-61 ожидает, что игрок продолжит клавишами управления
калькулятора, а программа затем сохранит значение, уже находящееся в X.
Последовательные присваивания `entered()` моделируют общий idiom ручных шагов:
`a = entered(); b = entered(); c = entered()` вводится как
`A ПП B ПП C С/П`, где каждый `ПП` выполняет ещё один `store`, а финальный
`С/П` продолжает программу. Обычный ввод хода всё ещё должен использовать
`read()`, который эмитит или сливает видимый input stop.

Если исходник сохраняет random seed, обновляет тот же seed через `random()`, а
затем использует и старый, и новый seed в одном чистом выражении, компилятор
может оставить предыдущий seed в стеке вместо сохранения и recall. Это
соответствует как прямому update `seed = random()`, так и однооператорному
helper, выполняющему этот update; имя helper не имеет значения.

`pow(base, exponent)` понижается в команду MK-61 `F x^y`. `bit_mask(index)`
строит packed bit mask с нулевым индексом: индекс `0` — это `1`, `1` — `2`,
`2` — `4`, `3` — `8`, а `4` начинает следующий разряд мантиссы как `10`.
`bit_has`, `cell_has` и соответствующие helpers `clear`, `set` и `toggle`
понижаются в синие логические операции, когда это выгодно. Helpers `has`
возвращают ноль или совпавшую маску, поэтому в тестах их следует сравнивать с
нулём. `bit_and(left, right)` и `bit_not(value)` также поддерживаются и
понижаются в побитовые логические операции (`К ∧`, `К ИНВ`) в том же семействе
выражений. `bit_or(left, right)` и `bit_xor(left, right)` тоже поддерживаются.
Одноаргументные математические и трансцендентные helpers включают `abs`,
`sign`, `int`, `frac`, `sqr`, `inv`, `sqrt`, `lg`, `ln`, `exp`, `sin`, `cos`,
`tg`, `asin`, `acos`, `atg`, `to_min`, `to_sec`, `from_sec` и `from_min`.
`sqr(x)`, `x * x` и `pow(x, 2)` понижаются через выделенный opcode MK-61
`F x^2`, когда исходное выражение можно безопасно вычислить один раз.
`pow(base, exponent)` — явная двухаргументная степень. `pow(10, exponent)`
использует выделенный opcode MK-61 `F 10^x`, как и `pow10(exponent)`.
`pow10(k)` доступен как удобный helper для десятичного масштабирования разрядов
и используется в логике досок и packed-строк во всех примерах. Константные
десятичные степени вроде `pow10(4)` и литерала `10000` материализуются через
MK-61 `F 10^x`, когда это короче ввода цифр. `sum(a, b, ...)` — вариадический
сахар для сложения. Он понижается как обычная цепочка `+`, но optimizer passes
могут распознавать его как явную accumulator-форму. `1 / x` понижается через opcode
обратной величины MK-61 (`F 1/x`), как и `inv(x)`. `max(a, b)` —
двухаргументный comparator helper (`К max`). `min(a, b)` доступен как
соответствующий helper `min-via-max`.

> **Особенность нуля в `К max`.** `max`/`min` напрямую отображаются на аппаратный
> opcode `К max`, который считает `0` **наибольшим** значением: если любой из
> операндов равен ровно `0`, результатом будет `0`. Поэтому `max(5, 0)` даёт
> `0`, а не `5`, и основанный на отрицании `min(-5, 0)` даёт `0`, а не `-5`.
> Это верно для реального поведения MK-61 (именно на это опираются оригинальные
> игровые листинги), и constant-folder повторяет эту семантику. Не используйте
> `max`/`min` для clamp-выражений вроде `max(coord, 1)` или
> `max(value, 0.0000001)`, если переменный операнд может быть ровно `0`.
>
> Когда нужен математически корректный результат, используйте `safe_max(a, b)`
> и `safe_min(a, b)`. Они полностью обходят `К max`, понижаясь через
> арифметические тождества `(a + b + |a - b|) / 2` и
> `(a + b - |a - b|) / 2`, поэтому `safe_max(5, 0)` равно `5`. Поскольку
> тождество ссылается на каждый операнд дважды, оба аргумента должны быть
> **дублируемыми (чистыми) выражениями** — идентификаторами, литералами или
> арифметикой над ними. Передача вызова с побочными эффектами (например
> `random()` или `read()`) отклоняется диагностикой; сначала привяжите его к
> переменной. Safe-helpers стоят на несколько ячеек дороже `К max`, поэтому они
> opt-in, а не default.
>
> Компилятор подкрепляет это тихим lint: когда объявленный диапазон
> нелитерального операнда `max`/`min` доказуемо включает `0` (например
> `counter 0..9`), в `report.warnings` добавляется заметка с предложением
> `safe_max`/`safe_min` (канал explain, рядом с raw-block hazards). Литеральный
> операнд `0` остаётся без предупреждения — тесты вида `max(x, 0) == 0` являются
> классическим намеренным idiom.

`pi()` — нульаргументный helper (`F pi`). `e()` эмитит `1; F e^x`, самый короткий
выделенный путь MK-61 для числа Эйлера.

Packed digit helpers используют индексы от единицы справа: `digit_at(value, 1)`
— разряд единиц, `digit_at(value, 2)` — разряд десятков. `digit_add` добавляет
цифру в эту позицию, а `digit_set` сначала удаляет старую цифру в этой позиции,
а затем вставляет новую.

Для source-shaped четырёхстрочных регистров, где разряд единиц зарезервирован,
а line-цифры начинаются с `10^1`, `packed_digit(value, index)` читает через
`frac(value / pow10(index))`, `packed_add(value, index, delta)` добавляет
`delta * pow10(index)`, а `packed_score(value, index)` применяет source-style
квадратичное отклонение, используемое компактными packed-line играми.

Helpers малых координатных множеств — обычные макросы выражений.
`near_any(value, radius, a, b, ...) >= 0` означает, что хотя бы один кандидат
находится в пределах `radius`; `near_any(...) < 0` означает, что ни один не
находится. `eq_any(value, a, b, ...) == 0` означает, что значение равно хотя бы
одному кандидату; `eq_any(...) != 0` означает, что оно не равно ни одному.

## Raw-блоки

Используйте `raw` только когда программе действительно нужна последовательность
калькуляторных команд, для которой ещё нет семантического helper. Raw-блок
разрешён внутри `loop`, `fn`, `if` и тел `match`, но должен объявлять свой
контракт:

```mkpro
fn invert_sum() {
  raw {
    takes Y = left, X = right
    returns X -> result
    clobbers X, Y, X1
    preserves state
    code {
      +
      К ИНВ
    }
  }
}
```

`takes` загружает стековые входы перед raw-кодом. Стековые входы должны быть
смежными от `X`: `Y` требует `X`, `Z` требует `Y` и `X`, а `T` требует `Z`, `Y`
и `X`. Компилятор загружает их снизу вверх, поэтому `takes Y = left, X = right`
входит в raw-код с `X = right` и `Y = left`.

`returns X -> name` сохраняет финальное значение `X` в объявленное поле
состояния после raw-кода. `clobbers` фиксирует побочные эффекты по стеку,
регистрам и дисплею, которых ожидает автор. `preserves state` обязателен:
raw-код не должен напрямую менять compiler-owned регистры состояния. Возвращайте
высокоуровневые значения через `returns`, а не угадыванием распределения
регистров.

Внутри `code` строки являются командами MK-61 или двухзначными hex-опкодами,
которые понимает raw instruction parser, плюс локальными метками вида `label:`.
Неизвестные raw-команды в контрактных raw-блоках — ошибки компиляции.
Raw-инструкции являются барьером оптимизатора для значений, которых они касаются;
отчёт записывает контракт и помечает raw-ячейки как константы, читаемые по
байтам.

Raw parser принимает весь диапазон опкодов `00`..`FF` как двухзначный hex. Он
также принимает документированные имена команд и распространённую нотацию из
справочника команд Анварова:

```mkpro
code {
  F π
  F√
  F x^{2}
  F x^{y}
  F↻
  ←→
  К°→′
  К°→′"
  К°←′"
  К∣x∣
  К∧
  К∨
  К⊕
}
```

ASCII-записи эквивалентны там, где они понятнее в source control:
`F pi`, `F sqrt`, `F x^2`, `F x^y`, `F reverse`, `<->`, `К °->′`,
`К °<-′"`, `К |x|`, `К *` и `К /`. Алиасы сравнений тоже нормализуются:
`F x!=0` можно писать как `F x≠0`, а `F x>=0` — как `F x≥0`.

Raw-команды ветвления принимают следующую метку или двухзначный адрес:

```mkpro
code {
  F x≠0 retry
  БП done
retry:
  К БП 7
done:
  С/П
}
```

Команды памяти принимают канонические и компактные формы. `X->П 3`, `X→П R3` и
`хП3` — одна и та же прямая запись; `П->X 3`, `П→X R3` и `Пх3` — один и тот же
recall. Косвенные формы используют синий префикс, например `К БП 7`,
`К x≥0 e`, `К X→П R4`, либо компактные `КБП7`, `КхП4`, `КПх4`.

Для недокументированных алиасов и обычно не вводимых команд используйте
hex-opcode напрямую: `1F`, `2F`, `3D`, `3E`, `4F`, `5F`, `6F`, `7F`..`EF` и
`F0`..`FF`. Каталог команд и машинные заметки см. в
`docs/03-command-reference.md`.

Когда ручной листинг содержит поддерживаемый обычно не вводимый байт `F*`, отчёт
может включать `# Patch Listing`. Введите основной листинг с показанным
placeholder `К НОП`, затем нажмите patch-клавиши, чтобы перезаписать этот адрес
финальным байтом. `--out keys` выводит ту же patch-последовательность даже когда
скомпилированный режим доставки — `hex`, потому что вывод клавиш по своей
природе является ручной процедурой.

## Диспетчеризация событий

Используйте `match`, когда код тайла или события выбирает одно из нескольких
действий. Если case требует несколько statements, вынесите их в функцию и
вызовите её из case:

```mkpro
match tile {
  0 => cave()
  3 => event_3(tile)
}

fn event_3(question) {
  memory_code = question
  warning()
  memory()
  answer = read()
  if answer == memory_code {
    strength++
    score++
    plans -= pos
  }
  else {
    strength -= 3
  }
}
```

Автор пишет обычный поток управления. Компилятор всё равно выбирает раскладку
диспетчеризации, инлайнинг, разделение хвостов и удаление ветвлений.

## Контракт оптимизатора

Модель машины — `mk61`. Компилятор всегда запускает максимальный оптимизатор и
автоматически рассматривает планирование стека, косвенный поток управления,
`FL0`..`FL3`, арифметическое удаление ветвлений, слияние хвостов, `В/О` как
одноклеточный `БП 01`, наложение кода и данных, адресные константы, тёмные
входы, алиасы R0/T, упаковку X2/display-byte, hex/sign формы мантиссы, no-op'ы
`F0`..`FF` и идиомы error-stop. Опкод `5F` моделируется отдельно от заполняющих
no-op'ов: в `mk61` это raw-преобразование состояния дисплея, а не зависание и
не универсальный заполнитель.

Оптимизатор автоматический. Исходный код никогда не говорит «используй тёмный
вход» или «используй X2»; компилятор либо доказывает предусловие и применяет
более короткое понижение, либо записывает более короткий кандидат как
отклонённый из-за отсутствующего доказательства или факта раскладки.

Исходник задаёт только то, что разрешено семантически. Отчёт объясняет:

- реально использованные машинные возможности,
- статические доказательства и допущения,
- факты эмулятора из модели MK-61,
- отклонённые более короткие кандидаты,
- внешний бюджет и горячие блоки.

Неподдержанные высокоуровневые эффекты вызывают ошибку компиляции, а не
становятся комментариями. Рабочего пути через spatial templates нет:
spatial-программы должны понижаться из своих реальных AST-узлов через обычный
IR. Запросы к доскам/клеткам понижаются в тот же IR выражений и потока управления,
что и остальной язык.

`reference` — только метаданные отчёта, они не должны менять генерацию кода.
Для известных references `games/*` отчёт находит исходный листинг, считает
реальный адресуемый span (`max(address)+1`), занятые entries и gaps, и сохраняет
`referenceSteps` равным этому span для существующих потребителей отчёта.
Отсутствующие листинги откатываются к бюджету с предупреждением.

Для скомпилированных программ отчёт помечает выбранные тактики, а не переключатели
исходника: косвенный поток через регистры, наложение кода/данных, X2/`ВП`,
упаковка display-byte и raw machine facts появляются только когда их выбрал
реальный IR-lowering или проход оптимизатора. Super-dark dispatch и cyclic/dark
layout остаются отклонёнными кандидатами, пока verifier раскладки не докажет обе
половины: ячейки входа/продолжения FA..FF и регистр диспетчеризации с доказанным
FA..FF selector. Понижение numeric dispatch обычно переупорядочивает варианты
ради самой дешёвой остаточной цепочки сравнений, но компилятор также пробует
size-rescue вариант в порядке исходника; решения распределителя для временного
ввода и scratch-регистров диспетчеризации принимаются в том же выбранном режиме
упорядочивания, что и понижение, поэтому спекулятивные кандидаты раскладки не
теряют обязательное хранение селектора.

Документированные capabilities вроде `branch-removal`, `arithmetic-if-*`,
`zero-condition-test`, `dispatch-compare-chain` и `fl-decrement-branch`
отмечаются как `active`, когда их буквальное rewrite сработало. Это привязывает
capability report к proof-backed поведению оптимизатора, а не к template paths.

## Текущие универсальные узлы исходника

Парсер сохраняет эти высокоуровневые statements как типизированные source nodes:

- `show(...)`
- `preview(expr)`
- `name = read()`
- `halt()`, `halt(expr)` или `halt("text")`
- `name = expr`
- `name += expr`
- `name -= expr`
- `name++`
- `name--`
- `if predicate { ... }`, `unless predicate { ... }`,
  `if predicate { ... } else { ... }` и
  `if predicate { ... } else if predicate { ... } else { ... }`.
  `else` можно писать либо на отдельной строке после `}`, либо как `} else {`.
  Predicates поддерживают `==`, `!=`, `<`, `<=`, `>`, `>=` и membership
  `item in list`. `unless predicate { ... }` — обратная форма
  `if predicate { ... }`: блок выполняется, когда predicate ложен.
  Голый predicate — сокращение сравнения с нулём: `if lives { ... }` означает
  `if lives != 0 { ... }`, `unless lives { ... }` означает
  `unless lives != 0 { ... }`, а `while lives { ... }` означает
  `while lives != 0 { ... }`.
- `while predicate { ... }`
- `loop { ... }`
- `fn_name(arg1, arg2)`
- `match expr { values => action }`. Case values разделяются запятыми и могут
  включать включительные integer ranges: `1..3 => action` то же самое, что
  `1, 2, 3 => action`. Case action — либо один inline statement, либо блок в
  фигурных скобках:

```mkpro
match key {
  1..3 => {
    score += 10
    lives -= 1
  }
  4 => score += 1
  otherwise => lives -= 1
}
```

  Блоки остаются в области видимости окружающей функции, поэтому `return`
  внутри block case возвращает из окружающей `fn`. Single-action cases и
  helper-функции (`1 => big_win()`) продолжают работать без изменений и
  понижаются тем же способом.
- query expressions: `line_count(set, cell)`, `neighbor_count(set, cell)` и
  `cell_at(board, pos)`
- formula helpers: `abs`, `acos`, `asin`, `atg`, `bit_and`, `bit_clear`,
  `bit_has`, `bit_mask`, `bit_not`, `bit_or`, `bit_set`, `bit_toggle`,
  `bit_xor`, `cell_at`, `cell_clear`, `cell_has`, `cell_mask`, `cell_set`,
  `cell_toggle`, `cos`, `digit_add`, `digit_at`, `digit_set`, `entered`,
  `eq_any`, `exp`, `frac`, `from_min`, `from_sec`, `inv`, `int`, `lg`, `ln`,
  `line_count`, `max`, `neighbor_count`, `near_any`, `packed_add`,
  `packed_digit`, `packed_score`, `pi`, `pow`, `pow10`, `random`, `sign`,
  `sin`, `sqr`, `sqrt`, `sum`, `tg`, `to_min`, `to_sec`.
- контрактные блоки `raw { ... }` для явных последовательностей команд MK-61

Присваивания, обновления, predicates сравнения, параметры функций, циклы и
вызовы функций понижаются через общий путь компилятора. Query expressions
захватываются как spatial query nodes до генерации кода. `loop` — реальный цикл,
а блоки `fn`
компилируются как процедуры `ПП`/`В/О`, прямые terminal jumps или inline-код в
зависимости от модели стоимости компилятора и анализа завершения. Вызовы должны
передавать ровно те параметры, которые объявлены у функции.

```mkpro
fn jump_to(floor) {
  current_floor = floor
  strength -= floor
}

fn jump_floor() {
  match current_floor {
    1 => jump_to(2)
    2 => jump_to(3)
    3 => jump_to(1)
  }
}
```

Когда все вызовы маленькой параметризованной функции используют литеральные
значения, а дублирование тела дешевле сохранения параметра и вызова общей
подпрограммы, понижатель автоматически специализирует такие вызовы.

Statement-level actions используют ту же форму вызова, что и остальной язык:
пишите `apply_step(player + 1)`, потому что `apply_step` — function call, а
`player + 1` — expression. Синтаксические слова и встроенные имена вроде
`show`, `read` и `halt` нельзя использовать как имена функций. Полный список
зарезервированных parser'ом имён функций:
`else`, `fn`, `halt`, `if`, `loop`, `match`, `otherwise`, `program`, `read`,
`requires`, `return`, `show` и `state`.

Текущие scalar lowerings всё ещё намеренно малы и проверяемы:

- `coord` представляется как packed numeric value; `pos.floor` обычно понижается
  в `int(pos / 100)`, причём `100` автоматически помещается в setup state, если
  это дешевле ввода трёх цифр. Для координат пещер `packed_decimal_zero_run`
  floor — это целая часть, поэтому `pos.floor` понижается в `int(pos)`.
- `random()` — это `[0, 1)` через `К СЧ`; `1` никогда не появляется, а `0` —
  только в редких hexadecimal-`Y` stack cases.
- `random(max)` — синтаксический сахар для `random() * max`,
  `random(min, max)` — синтаксический сахар для
  `min + random() * (max - min)`, а `random(domain)` выбирает координату из
  домена доски. Числовые верхние границы исключительные. `int(random(max))` и
  `int(random(min, max))` с целочисленным нижним пределом берут floor
  масштабированной выборки через `x - frac(x)` перед добавлением смещения,
  избегая `К [x]` сразу после `К СЧ`.
- `cells(domain) = random()` переносится оптимизатором в setup-состояние,
  отражённое в отчёте.
- Принадлежность набору клеток записывается как `item in collection`;
  понижение сохраняет компактную форму сравнения внутренне, когда это меньше.
- Награды — обычные обновления вроде `treasure += expr`.

## Справочник по оптимизации

Этот справочник языка намеренно заканчивается на контракте оптимизатора.
Конкретный каталог переписываний, capability IDs, модель доказательств и порядок
IR-pass'ов до неподвижной точки поддерживаются в
[MK-Pro Optimization and Lowering Strategy Reference](20-mkpro-optimization-reference.md).

Для авторов исходников практические правила такие:

- Пишите семантические конструкции MK-Pro (`show`, `read`, `fn`, `match`,
  `loop`, обновления состояния и `raw`), а не подсказки размещения на машине.
- Компилятор может сливать запросы ввода, inline'ить или разделять функции,
  менять раскладку диспетчеризации, переиспользовать состояние стека/регистров,
  активировать формы X2/display-byte и выбирать косвенные или зависящие от
  раскладки машинные тактики, когда есть доказательства, попадающие в отчёт.
- `mk-pro explain` и `mk-pro --out json` — источник истины о том, какие
  оптимизации сработали, какие машинные возможности были использованы и почему
  более короткие кандидаты были отклонены.

Контракт языка состоит в том, что эти преобразования сохраняют наблюдаемое
поведение MK-61 в выбранной модели точной машины. Неподдержанные
высокоуровневые эффекты всё равно приводят к ошибке компиляции, а не становятся
комментариями или неявными машинными шаблонами.
